# Document Delivery Pipeline Specification

This document details the technical specification for our document generation and email dispatch systems. It ensures future engineers and agents understand how personalized meal plan data is assembled, converted into PDFs via Gotenberg, persisted to R2, and delivered securely to users via Brevo.

## 1. GOTENBERG PDF RENDERING CONTEXT

The PDF generation process converts structured JSON payloads into highly customized physical documents.

- **Payload & React HTML Translation**: The worker receives a structured payload containing `Identity`, `MetabolicProfile`, `Meals`, and AI `IntelligenceNotes`. Before rendering, `normalizePayload()` standardizes values (e.g., mapping `cut` to `Fat Loss`) and filters out empty meals. The data is then rendered via `react-dom/server` (`renderToStaticMarkup`) into a sequence of static HTML pages (`Page1`, `Page2`, `NotesPage`, etc.).
- **Gotenberg Interaction**: The final HTML blob is wrapped in a strict A4 template layout and injected into a `FormData` object. The worker makes an HTTP POST request to our Gotenberg instance (`/forms/chromium/convert/html`).
- **Authentication**: Basic Auth (`worker:beta_render_token_491`).
- **Resilience**: Because Gotenberg is hosted on Fly.io and is subject to cold starts, the network call is wrapped in a `withRetry` block configured for **3 retries with a 2000ms backoff**.

## 2. CLOUDFLARE R2 OBJECT STORAGE DEPLOYMENT

Once Gotenberg returns the PDF as a `ReadableStream`, it is immediately streamed into Cloudflare R2 for persistent storage.

- **Binding Configuration**: `PDF_STORAGE`
- **Key Naming Conventions**: The standard schema for storing objects is `pdfs/{jobId}.pdf`. During retrieval, the `getPdfFromR2` function attempts to fetch `pdfs/{jobId}.pdf` first, with a fallback to a bare `{jobId}` key to ensure compatibility with our older PDF Orchestrator workflows.
- **Upload Details**: The file is streamed directly to R2 using `env.PDF_STORAGE.put()` with HTTP metadata set to `contentType: "application/pdf"`. Custom metadata (`jobId`, `uploadedAt`) is tagged on the object.
- **Delivery Vector**: We do not generate time-limited pre-signed URLs from R2. Instead, the frontend and email templates use a worker-proxied route (`/api/download/{jobId}`) which abstracts the R2 lookup securely.

## 3. BREVO EMAIL TRANSMISSION INTEGRATION

Our transactional email pipeline handles the final step of delivering the generated asset to the user.

- **Authentication**: API calls are made directly to Brevo's REST API (`https://api.brevo.com/v3/smtp/email`) using the `BREVO_API_KEY` header.
- **Payload Structure**: The system does **not** rely on Brevo UI templates. Instead, the HTML content is dynamically generated by our worker (`renderEmail(fullName, downloadUrl)`).
- **Attachment Strategy**: To maximize email deliverability and avoid spam filters, the PDF is **not attached** directly to the email payload. Instead, the secure worker-proxied download link (`downloadUrl`) is embedded within the email HTML.
- **Resilience**: The Brevo dispatch is wrapped in a `withRetry` block, allowing up to **3 retries with a 1000ms backoff** in case of transient API failures.

## 4. ERROR HANDSHAKES & BOUNDARIES

The document pipeline is architected to fail gracefully, ensuring worker threads do not hang indefinitely:

- **Gotenberg Failures (`PDF_GEN_FAIL`)**: If rendering fails after all 3 retries, an explicit exception is thrown with the Gotenberg HTTP response body. This halts the worker pipeline, triggering a status update to `failed` in the Ledger and logging the context to Sentry.
- **R2 Failures**: The R2 service `uploadPdfToR2` catches internal errors and returns `null` instead of throwing an exception, allowing the pipeline to decide how to handle missing storage keys (which ultimately trips the final delivery check).
- **Brevo Failures (`EMAIL_SEND_FAIL`)**: If the Brevo API rejects the payload, an exception is bubbled up containing the raw Brevo error text, ensuring the job status reflects an email delivery failure without corrupting the underlying PDF generation state.
